## `exp` is now single-argument

Old form `exp(G, s)` was removed. Parser at `parseExpression.js` matches `exp(<mv_expr>)` only â€” any expression that splits on a top-level comma is rejected.

`motorExp.compute`:

- Scalar input â†’ `Math.exp(s)`.
- Otherwise, evaluate the argument to an MV `V` and call ganja's `V.Exp()` instance method.

Earlier in the day this was a closed-form `cos(c) + (sin(c)/c)Â·V` branch (with `c = V[6]`) plus a separate nilpotent path `1 + V` when `VÂ² = 0`. The closed-form is exact only when VÂ² is a pure scalar â€” fine for pure rotors and pure translators in PGA(2,0,1), but wrong when V mixes e0 and e12. Delegating to `V.Exp()` covers the general case for free, so the branch was deleted.

## `V * blade` fix

`parseMVExpr` matches `var * blade` and stores it as a *scalar* coefficient in `coeffExprs`. When V is a vector or other multivector, `evalExpr('V', â€¦)` returns NaN, and the resulting MV had garbage in the blade slot.

`multivector.compute` now detects when a coefficient is a single var (optionally negated) that resolves to a *non-scalar* dep, and accumulates the true algebraic product `dep Ã— blade` instead. `V*e12` now returns `vxÂ·e01 + vyÂ·e02` (the perpendicular ideal point). Existing forms like `x*e01 + 2*e02` with scalar x still go through the original path unchanged.

## Feature audit fixes

Branch `fix/feature-audit`. Systematic sweep across constructors, operators, motors, normalization, interactivity, animation, theming, and showcase. Three latent bugs:

### 1. `.blade` accessor in scalar coord expressions

`point(P.e01, â€¦)` failed: `evalCoords` â†’ `evalExpr` (scalar-only) can't resolve `.blade` accessors. Added `evalScalar` in `evalExpr.js` that routes through `evalMVArith` when the expression contains `.<letter>`, else falls back to `evalExpr` (preserves PI/E etc.). Wired into freePoint/vector/freeLine coord eval and `resolveInlineGeom` + multivector coeff fallback.

### 2. Blade names leaking into deps

`extractVarNames` picked up `e01`, `e02`, â€¦ as identifiers, polluting freePoint/vector/freeLine deps. The graph engine then short-circuits because `values["e01"]` is undefined. Filtered via the now-exported `parseBladeName` from `evalMVArith.js`.

### 3. `!e01` / `~e12` mis-parsed

The `!ID` / `~ID` regex didn't exclude blade names, so `!e01` parsed as "dual of named node e01" instead of "dual of the e01 blade". Skipped those forms when `parseBladeName(id)` returns truthy, letting them fall through to `mvExpr` where the unary ops handle blades correctly.

### Known limitations left as-is

- `M >>> [P1, P2, ...]` doesn't transform polygons as a unit.
- Operator precedence is flat (single `parseTerm` level): `A * B ^ C` is `(A*B) ^ C` not the GA-conventional `A * (B ^ C)`. Use parens.

(A third item â€” `exp(V)` with mixed e0/e12 components â€” was *also* on the audit list as a limitation but became moot once `V.Exp()` took over.)

## Visual weight

`objectWeight(val)` in `pga.js` returns the scalar weight of a PGA object, and SVG renderers in `Canvas.jsx` multiply their stroke widths / point radii by it. So `5*e12` (a point with weight 5) renders 5Ã— thicker than `e12`; `5*(e1+e2)` is a 5âˆš2-thick line; `5*e0` is a 5Ã—-thick ideal line.

| Object form | Weight returned |
|---|---|
| Finite point (e12 â‰  0) | `\|e12\|` |
| Ideal point (e12 = 0) | `âˆš(e01Â² + e02Â²)` |
| Line thru origin (e0 = 0) | `âˆš(e1Â² + e2Â²)` |
| Line w/ offset (e0 â‰  0) | `\|c\|` via `PGA.Length(PGA.Dual(L))` |
| Ideal line (only e0) | `\|c\|` via Dual |
| `{vx, vy}` vector | Euclidean magnitude |

## Ideal-point / ideal-line rendering

`Canvas.jsx`:

- Ideal line (pure e0): drawn as a dashed ellipse inscribed in the canvas â€” the "line at infinity" boundary of the projective plane. Visual is in screen-space, doesn't move with pan/zoom.
- Ideal point markers: only drawn when *some* visible node *is* the ideal line. The point at infinity in direction `(vx, vy)` is plotted where that direction meets the ellipse boundary (`idealPointEllipsePos`).
