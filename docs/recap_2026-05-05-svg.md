# SVG Rendering Migration — 2026-05-05

## What changed

`Canvas.jsx` was rewritten from imperative **HTML5 Canvas 2D** rendering to declarative **SVG** (native React JSX). The change is purely in the rendering layer; all graph state, coordinate math, hit-testing, and drag logic are unchanged.

---

## Before (Canvas 2D)

- One `<canvas>` element with a `useEffect` draw loop running after every render
- Imperative functions: `drawGrid(ctx, vp, W, H)`, `drawPoint(ctx, ...)`, `drawLine(ctx, ...)`, `drawVector(ctx, ...)`
- Canvas needed explicit `width`/`height` attributes set via `ResizeObserver` for pixel-perfect rendering
- Lines clipped manually with `ctx.rect(); ctx.clip()`
- Mouse coordinates needed scaling: `(e.clientX - r.left) * (canvas.width / r.width)`

## After (SVG)

- One `<svg>` element with `overflow="hidden"`; React re-renders the SVG tree declaratively
- SVG component functions: `SvgGrid`, `SvgPoint`, `SvgLine`, `SvgVector`
- ResizeObserver still used, but only to track `size` state for coordinate math — no canvas attribute manipulation
- Lines clipped automatically by SVG viewport (`overflow="hidden"`)
- Mouse coordinates are direct CSS pixels: `e.clientX - r.left` (no scaling ratio)
- No explicit draw loop — rendering is a pure function of state

---

## Architecture

```
Canvas()
├── <div ref={wrapperRef}>          wrapper, ResizeObserver target
│   └── <svg ref={svgRef}>
│       ├── <rect>                  background fill (#fafafa)
│       ├── <SvgGrid>               grid lines + axis + labels
│       └── {orderedNodeIds.map()}  one SVG component per drawable node
│           ├── <SvgPoint>          <circle> + optional <text>
│           ├── <SvgLine>           <line> + optional <text>
│           └── <SvgVector>         <line> + <polygon> (arrowhead) + <circle> (tail)
```

### Coordinate system

All SVG elements are placed in **screen space** (CSS pixels). The same `w2c` / `c2w` functions from before map between world and screen:

```
screenX = offsetX + worldX × scale
screenY = offsetY − worldY × scale   ← Y-axis flip
```

No SVG `<g transform>` is used for the world; each component calls `w2c` individually. This avoids text-flip issues and keeps the code straightforward.

### Viewport / zoom / pan

State: `vp = { scale, offsetX, offsetY }` — unchanged from the Canvas 2D version.

- **Zoom**: `onWheel` on `<svg>` updates `vp.scale` and recenters on cursor
- **Pan**: `onPointerDown/Move/Up` on `<svg>` translate `vp.offsetX/Y`
- **Resize**: `ResizeObserver` shifts `offsetX/Y` to keep the geometric origin centered

### Hit-testing

Unchanged — the `hitTest()` function still works in screen space using `w2c`. SVG per-element pointer events are not used (the global `onPointerDown` on `<svg>` is the single entry point, same as before).

## Benefits of SVG

| | Canvas 2D (before) | SVG (now) |
|---|---|---|
| Rendering | Raster — blurry at high DPI without DPR scaling | Vector — crisp at any zoom / DPI / print |
| Code model | Imperative draw loop | Declarative JSX components |
| Draw loop | Explicit `useEffect` after every render | React re-renders the SVG tree automatically |
| Text | Manual `ctx.fillText` with coordinate clamp | `<text>` element, CSS-styleable |
| Clipping | Manual `ctx.rect/clip` | `overflow="hidden"` on `<svg>` |
| Export | Would need `canvas.toDataURL()` | Direct: `svgRef.current.outerHTML` → `.svg` file |
| Performance | Excellent for any object count | Excellent for <1000 objects (DOM overhead negligible here) |
